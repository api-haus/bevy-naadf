# Mobile GPU-binding-budget preselection — reuse audit

Goal: design a **startup-time GPU budget preselection routine** that reads
`device.limits()` and rewrites the four oversized storage-buffer footprints
(`voxels`, `blocks`, `taa_sample_accum`, `taa_samples`) — composed via three
levers (TAA ring depth, world-size-in-segments, internal-resolution scale) —
**before** the world install fires, so the full Naadf world boots on mobile
WebGPU (256 MiB `max_storage_buffer_binding_size`).

Worktree audited: `/mnt/archive4/DEV/bevy-naadf/.claude/worktrees/android-build`.

## Top reuse recommendation

The **`TaaRingConfig` mirror pattern** (`crates/bevy_naadf/src/render/mod.rs:105-118` + `crates/bevy_naadf/src/render/taa.rs:46-50` + `crates/bevy_naadf/src/render/pipelines.rs:363-365`) is the single highest-leverage piece of existing code: it is the **complete, working template** for "configure something on `AppArgs`, mirror it into the render sub-app at plugin-build, consume it from both Rust buffer sizing and WGSL shader-def, single source of truth." Lever #1 (TAA ring depth) is already wired top-to-bottom through this pattern — the budget routine for that lever is a one-line `args.taa_ring_depth = chosen_depth;` mutation before `build_app_with_args`. Lever #2 (world-size-in-segments) and lever #3 (internal-resolution scale, currently absent — derived from window size at `crates/bevy_naadf/src/render/extract.rs:357`) should both be designed to copy this exact shape: store the override on `AppArgs`, mirror at plugin build, consume from the existing allocator + shader-def sites. The `ConstructionConfig::from(&AppArgs)` wasm32 clamp at `crates/bevy_naadf/src/render/construction/config.rs:265-287` is the prior art for the platform-conditional override that selects the values — same `cfg(target_os = "android")` + (future) `--mobile` flag layering.

The actual `device.limits()` reading site already exists twice (`render/prepare/world.rs:391` + `render/construction/validation.rs:1048`), so the pattern of "extract `RenderDevice`, call `.limits()`, compare per-binding bytes against `max_storage_buffer_binding_size`" is settled idiom in this codebase — copy the math, not the timing (both current sites read post-install; the new routine reads pre-install).

## Borderline calls

- **`ConstructionConfig::from(&AppArgs)` — extend vs not applicable.** The shape is right (a single `From<&AppArgs>` lift, platform-conditional clamp inside), but it runs inside `ConstructionPlugin::build` (`crates/bevy_naadf/src/render/construction/mod.rs:1832-1836`), which fires *during* `build_app_with_args`, *after* `AppArgs` is locked in. The budget routine must run *before* `build_app_with_args` consumes `AppArgs`. Flips to firmly "reuse" if the orchestrator decides the budget routine writes into the render sub-app *only* and leaves main-world `AppArgs` untouched — which is the wrong design for lever #2 (`WORLD_SIZE_IN_SEGMENTS`) because the main-world `setup_test_grid` reads `WORLD_SIZE_IN_CHUNKS` at `voxel/grid.rs:187, :245-250` **before** the render world even sees `ConstructionConfig`.

- **`validate_gpu_construction_production_scale` — extend vs not applicable.** This routine spins up its own headless render world via `App::new() + DefaultPlugins + Plugin::ready + Plugin::finish` and pulls `RenderDevice` out (the technique at `world/buffer.rs:246-264`). It could be the literal template for the pre-install limits read — but only if the orchestrator accepts a "spin up a throwaway render adapter to probe limits, then teardown, then re-create the real App with budget overrides" architecture. Cheaper alternatives: (a) add a `Startup` system to the real App that runs before `setup_test_grid` and mutates `AppArgs`-backed budget resources in-place, but this fights Bevy's normal extract-flow because `RenderDevice` lives in the render sub-app and most main-world resources can't be mutated mid-Startup safely; (b) read limits via `wgpu::Instance::request_adapter` directly outside the Bevy app entirely. Flips to "reuse" if (a) probe-then-rebuild is acceptable; "not applicable" if the orchestrator picks (b).

- **`WORLD_SIZE_IN_SEGMENTS` migration surface — extend vs not applicable.** Counted 15 distinct consumer sites; most are trivial swaps to a `Res<WorldSize>` runtime resource, but `world_size.rs:26-40` is a `const fn` derivation tree (segments → chunks → voxels via `mul_uvec3`) that is currently bit-checked at compile time against the C# canonical numbers (`world_size.rs:46-54` test). Migrating to a runtime resource loses the compile-time pin. The two test sites at `voxel/grid.rs:1184, :1266` consume `WORLD_SIZE_IN_CHUNKS` as a target for static unit tests — they expect the C#-canonical 256×32×256 value. Flips to "not applicable" if the orchestrator decides the mobile profile keeps the const but adds a parallel "effective world size" runtime resource that diverges only when a mobile flag is on — leaves the C# faithful-port invariant intact and isolates the divergence.

## Side notes / observations / complaints

1. **Two separate limits-read sites already exist in this codebase, both written by previous diagnostic dispatches**: `render/prepare/world.rs:391` (in-line Q4 instrumentation) and `render/construction/validation.rs:1048` (top-level `--validate-gpu-construction-production`). Both share the pattern `limits.max_storage_buffer_binding_size / (1024 * 1024)`. The brief's headroom constant (`192 MiB = 75% × 256 MiB`) should live ONCE somewhere shared — recommend a new `crates/bevy_naadf/src/render/budget.rs` (or `mobile_budget.rs`) module that owns `MIN_STORAGE_BINDING_CAP_BYTES`, `MOBILE_HEADROOM_FACTOR`, and the per-lever ladder constants. The orchestrator should explicitly name this module's location in the design phase.

2. **Lever #3 (internal-resolution scale) does NOT exist yet as a config knob.** Searched: no `internal_res`, `render_scale`, `resolution_scale` field anywhere. The viewport size is derived inside `crates/bevy_naadf/src/render/extract.rs:357` via `camera.physical_viewport_size()` — directly off the window. Adding lever #3 means either (a) inserting a render-scale field into `AppArgs` and dividing `viewport_size` by it inside `extract_camera`, or (b) using Bevy 0.19's `Camera::sub_camera_view` / `Camera::viewport`. Of the three levers the brief composes, lever #3 is the only one with **no existing wiring**, so the design phase should treat it differently from #1 and #2.

3. **The brief mentions a `--probe` flag escape hatch.** No such flag exists in the repo today — the closest thing is the wholesale Android-only `android_main::main` (`android_main.rs:36-49`), which is a separate `#[bevy_main]` entry, not a CLI flag on the production binary. The CLI parse in `crates/bevy_naadf/src/main.rs:39-52` is hand-rolled (no `clap`), 13 lines total, currently handles only `--vox <path>`. Adding `--probe` is a 5-line extension; the e2e binary has the more elaborate dispatch (`crates/bevy_naadf/src/bin/e2e_render.rs:138-156`) but the production binary's flag surface is intentionally minimal. Recommend the orchestrator's design phase pick *which* binary gets the `--probe` flag (production-only? both?) early — adding it to `e2e_render` is free, adding it to `main.rs` widens the production-binary CLI surface.

4. **Shaders are already runtime-configurable for world size.** WGSL files at `crates/bevy_naadf/src/assets/shaders/bounds_calc.wgsl:78, :236-238, :261-262`, `chunk_calc.wgsl:72-73`, `entity_update.wgsl:78` all read world size from runtime uniform `params.size_in_chunks` — the Rust side fills these uniforms from `crate::WORLD_SIZE_IN_CHUNKS` (`producer.rs:230-247`, `validation.rs:887-900`). This is good news for lever #2: the migration is Rust-only; no shader rewrite needed. Bad news: the `crate::WORLD_SIZE_IN_VOXELS` access at `producer.rs:138-140` feeds into a per-segment generator-model uniform, so even after lever #2 is wired, the dispatch loop at `producer.rs:179-181` (`for sz in 0..crate::WORLD_SIZE_IN_SEGMENTS.z`) must read the runtime size, not the const.

5. **The `WORLD_SIZE_IN_*` derivation chain has a compile-time C#-canonical pin** (`crates/bevy_naadf/src/world_size.rs:46-54`) that asserts `WORLD_SIZE_IN_CHUNKS == UVec3::new(256, 32, 256)` and `WORLD_SIZE_IN_VOXELS == UVec3::new(4096, 512, 4096)`. The user's stated rule in `~/.claude/CLAUDE.md` is "no Bevy-only microoptimizations or behaviors not in C# NAADF; default = match C#, even when C# has the bug. Deliberate divergences require explicit user approval + docs entry." Mobile profile cuts the world to 128×32×128 or 64×32×64 chunks — this is a **deliberate C# divergence** and the design phase must surface the docs-entry requirement explicitly. The `master-branch-identity` memory ("aggressive deletion of non-C#-parity rot is encouraged on master") makes it doubly important to land this as a feature-flagged mobile profile, not a global default.

6. **`DEFAULT_TAA_RING_DEPTH` test in `app_args.rs:228-235`** pins the default to `matches!(depth, 16 | 24 | 32)`. The brief's TAA ladder `{32, 24, 16, 8, 4, 0}` introduces values *outside* the currently-supported set. This test will fail (and rightly so — silent corruption guard) the moment the budget routine picks 8 or 4. The design phase needs to either (a) widen the supported-values test to the full ladder, or (b) keep the default at 32 and only allow the runtime budget routine to override it (the test pins the *default*, not the runtime value — current test text matches case (b)). Worth deciding which up front.

7. **There is no `Startup` system that runs after the renderer is available but before world install.** The current `Startup` schedule pyramid (from grep at `voxel/plugin.rs:34`, `lib.rs:300`, `render/construction/mod.rs:1849-1858`, `dev_font.rs:42`) all fires during the same `Startup` schedule, with `.after()` constraints chaining individual systems. The render sub-app's `RenderStartup` (`render/mod.rs:147`, `render/pipelines.rs:350`) runs in the render world's own schedule after `RenderDevice` exists. Reading `device.limits()` before `setup_test_grid` runs is **non-trivial** because `setup_test_grid` is a main-world `Startup` system and `RenderDevice` lives in the render sub-app — main-world `Startup` cannot reach into the render sub-app's resources cleanly. The realistic insertion point is **before** `build_app_with_args` even starts, by spinning up a probe `App` to extract `RenderDevice` (the `validate_gpu_construction_production_scale` pattern at `validation.rs:1037-1046` + `world/buffer.rs:246-264`), reading limits, dropping the probe app, then building the real app with budget-overridden `AppArgs`. This is architectural news the design phase needs to confirm.
