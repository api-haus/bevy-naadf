# 00-reuse-audit — web-vox-color-divergence
2026-05-18

Read-only audit of the existing codebase landscape for the web-only color
divergence investigation. The audit maps install-paths, palette/material
upload sites, render-world extract systems, change-detection patterns,
diagnostic surfaces, and existing e2e gates that already capture
color-relevant data.

This file feeds the next diagnose-first agent. No fixes proposed; the
goal is "what is already there to reuse, extend, or instrument."

## Hypothesis index (referenced in the table below)

1. Parse/install split lost a side-effect (palette upload, ModelData
   mutation, Changed<T> trigger).
2. Resource-change detector not firing on async install (insert vs ResMut).
3. Default-scene palette interference (`setup_test_grid` Startup runs
   before async `.vox` lands).
4. Async install timing vs Q3 cross-frame readback state machine.
5. wgpu 29 / WebGPU binding shape divergence.
6. Lighting/exposure: albedo is zero but lighting math still applies.

## Top reuse recommendation

The single most load-bearing finding is **the build-once gate on `stage_world_gpu_buildonce` (`extract.rs:201-203`) + `prepare_world_gpu` (`prepare.rs:201-203`)** combined with the fact that **`setup_test_grid` unconditionally installs the default-scene `VoxelTypes` at Startup** (`grid.rs:104-143` → `:220-313`). On web, the rayon parse completes N frames after Startup → `install_imported_vox` calls `commands.insert_resource(VoxelTypes { types: imp.palette })` (`grid.rs:580`) → but the extract+prepare chain is permanently gated off. The `naadf_world_bind_group` reads from a `GrowableBuffer<GpuVoxelType>` that was uploaded ONCE (`prepare.rs:487`) with the default palette. Native gates don't hit this because `setup_test_grid`'s `GridPreset::Vox { path }` arm calls `install_vox_in_fixed_world` synchronously — `WorldData + VoxelTypes` with the `.vox` palette are inserted BEFORE the first `ExtractSchedule` runs.

**Reuse + extend**: `stage_world_gpu_buildonce` and `prepare_world_gpu` are the two systems whose gating logic must change (or be supplemented by a `Changed<VoxelTypes>` / `Changed<WorldData>` re-build path). The shape of the fix is "re-extract + re-prepare when the source resources changed" — but the audit's job is to surface this, not propose it. The supporting evidence: **zero `Changed<VoxelTypes>` queries in the codebase** + **`commands.remove_resource::<WorldGpu>()` does not exist** + **`stage_world_gpu_buildonce`'s own docstring (`extract.rs:64-66`) explicitly acknowledges "If a future feature ever needs a whole-world re-upload (e.g. world reload or live re-import), it re-creates this resource at that boundary — but no such code path exists today."**

For diagnostic instrumentation, the highest-value reuse points are: (a) add an `info!` log at `prepare.rs:487` reporting palette signature (first-N entries' `color_base` channels) so the upload event is visible; (b) extend `assert_vox_geometry_visible` in `vox_e2e.rs:402` from luminance-only to per-channel spread (the `region_mean` already returns `[f32; 4]`, only the assertion needs widening); (c) extend `--vox-web-parity` to also assert per-channel max ≥ floor on the loaded frame, since the current SSIM-only assertion would PASS on a structurally-correct but all-dark render.

## Borderline calls

- **`populate_cpu_mirror_from_gpu_producer` + the Q3 readback state machine** (`mod.rs:1028-1325`) — verdict **reuse** because the readback only handles geometry (chunks/blocks/voxels), NOT palette. But borderline because the brief explicitly cites this as a hypothesis-4 site. What would flip the verdict to **extend**: if instrumentation shows the readback completes successfully (no `ReadbackStage` stalls) and the geometry is correct in the framebuffer (silhouettes visible, just wrong color), Q3 is definitively NOT the cause and this candidate stays at "reuse". If the readback stalls past frame 600 on web and the framebuffer is dark — flip to "extend" to investigate target-specific readback timing.

- **`stage_model_data_buildonce` (`extract.rs:242-259`)** — verdict **reuse** because the default scene doesn't insert `ModelData`, so the build-once gate is open when the async install lands. But borderline because the parallel symmetry with `stage_world_gpu_buildonce` is suspicious. What would flip the verdict to **extend**: if instrumentation shows `ModelDataRender` IS populated correctly (`size_in_chunks` matches the .vox, `data_chunk/block/voxel` lengths match) and the geometry renders correctly — confirms ModelData extract works fine, palette is the diverging axis. If `ModelDataRender` is also stuck at empty/default — both extracts share a common gate-bug.

- **`--vox-web-parity` gate (SSIM 0.0175)** — verdict **extend** rather than **reuse** because the existing SSIM assertion passes today on a regression that produces "geometry-correct but color-wrong" frames. The dissimilarity assertion only catches "is the scene empty" not "are the colors meaningful". What would flip it to a pure **reuse**: if a follow-on instrumented run confirms the loaded frame has near-zero per-channel means (R≈G≈B≈10/255), the SSIM compare already correctly flags structural mismatch from skybox baseline — but no, even a dark voxel scene has VERY different structure from a gradient sky, so SSIM stays low ≈ 0.02. **The gate is genuinely color-blind today.** Verdict stays **extend**.

- **`install_imported_vox`'s `commands.insert_resource(...)` discipline** (`grid.rs:545-580`) — verdict **reuse + instrument**. Borderline because Bevy's `commands.insert_resource()` *should* trigger `is_changed()` for one-tick `Changed<R>` queries (per Bevy semantics), but no system uses `Changed<>` on these resources. What would flip to **extend**: if Bevy 0.19's command-buffer semantics on `insert_resource` are confirmed to NOT trip change-detection when the resource already exists (a known Bevy gotcha — `insert_resource` over an existing resource does set `Changed` per Bevy's own docs, but the trace is worth verifying), this candidate becomes the site to add `Removed<WorldGpu>()` followed by `insert(WorldGpuStaging)` to force a rebuild.

- **`install_default_embedded_in_fixed_world` running unconditionally at Startup on web** (`grid.rs:104-143`, `lib.rs:758-791`) — verdict **reuse** (root-cause site, not new code). Borderline because one reading is "the default scene should NOT run on web when an async .vox is incoming" — i.e. add a `WebAsyncVoxPending` resource to suppress it. What would flip to **extend**: if the team's chosen fix path is "suppress default scene during async fetch" rather than "make extract/prepare re-buildable on resource change", this candidate becomes the extension site (a new `Option<Res<WebAsyncVoxPending>>` short-circuit in `setup_test_grid`).
