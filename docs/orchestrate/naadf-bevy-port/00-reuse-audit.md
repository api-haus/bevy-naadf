# Reuse Audit — Porting NAADF to Bevy

**Date:** 2026-05-14
**Auditor:** re-implementation auditor (delegated orchestration)
**Goal:** Port `/mnt/archive4/DEV/NAADF` (C#/MonoGame voxel GI engine, "Nested Axis-Aligned Distance Fields", CGF 2026) to Rust/Bevy in `/mnt/archive4/DEV/bevy-naadf`.
**Target audited:** `/mnt/archive4/DEV/bevy-naadf` — every file under `src/`, plus `Cargo.toml`, `README.md`.
**Source skimmed (breadth-first):** `/mnt/archive4/DEV/NAADF/NAADF` subsystem tree; research doc `docs/research/ulschmid-2026-naadf-voxel-gi.md` (headings only).

## Summary

The `bevy-naadf` target is a **toolchain proof-of-concept**, not a partial port. It stands up Bevy 0.19-rc.1 + Solari raytracing + DLSS Ray Reconstruction against a *generic* procedural Cornell-box-style scene. There is **no voxel code, no AADF data structure, no chunk/world subsystem, and no NAADF-specific rendering** in the target yet. The reusable assets are the *infrastructure scaffolding* (app wiring, camera, HUD, DLSS toggle, build setup) and — critically — the **decision already baked in to lean on Bevy Solari as the raytracing/GI substrate** rather than porting NAADF's hand-written HLSL ray pipeline. Per the README roadmap, milestone 1 (this scaffold) is done; milestones 2 (voxel data structures + AADF compute) and 3 (DLSS into the NAADF renderer) are greenfield.

## NAADF subsystems still entirely unported (greenfield)

None of the following exist in `bevy-naadf` in any form — they are the actual port surface:

- **Voxel core library** — `Libraries/VoxelsCore/` (`VoxelData`, `VoxelDataBytes`, `VoxelDataColors`, `Voxel`, `Material`, `Color`, `XYZ`, `BoundsXYZ`, `MagicaVoxel`/`VoxFile`/`Voxlap` importers, `VoxelImport`). Grid types, voxel/material model, file-format importers. Bundled `obj2voxel.exe` is an external tool — needs a Rust equivalent or subprocess wrapper.
- **AADF / multi-layered cell data structure** — the paper's core contribution (research doc §3.1–3.3: multi-layered cell structure, construction, augmenting cells with AADFs). **No counterpart anywhere** — this is the heart of the port and the hardest greenfield piece.
- **World data subsystem** — `World/Data/` (`WorldData` 522 lines, `ChangeHandler`, `BlockHashingHandler`, `WorldBoundHandler`, `EntityData`/`EntityHandler`, `EditingHandler` + `EditingTools/` cube/sphere/paint/floodfill/model). Chunk/block storage, GPU buffer sync, edits, dynamic entities (research doc §3.5–3.6).
- **World generation** — `World/Generator/` (`WorldGenerator`, `WorldGeneratorModel`) + shaders `world/generator/generatorModel.fx`. GPU-driven world gen into chunk buffers/3D textures.
- **Model subsystem** — `World/Model/` (`ModelData`, `ModelHandler`) + `world/model/typeMapping.fx`. Voxel model placement/instancing.
- **Voxel type / material system** — `World/VoxelTypeHandler.cs` (169 lines; `MaterialTypeBase`/`MaterialTypeLayer` enums — Diffuse/Emissive/MetallicRough/MetallicMirror). Maps to Bevy materials but the layered-material model is NAADF-specific.
- **World render pipeline** — `World/Render/` (`WorldRender`, `Versions/WorldRender{Albedo,Base,PathTracer}`, `Atmosphere`) + the entire `Content/shaders/render/` HLSL tree. *Partially superseded by Solari* (see table) but the voxel-traversal first-hit, atmosphere model, and AADF-specific traversal (research doc §3.4) still need bespoke Rust/WGSL.
- **Long-term-memory TAA & resampling pipeline** — research doc §4.1–4.3 (`commonTaa.fxh`, `renderTaaSampleReverse.fx`, `renderSpatialResampling.fx`, `renderSampleRefine.fx`, `renderDenoiseSplit.fx`). Overlaps with Solari ReSTIR + DLSS-RR — needs an explicit map of which passes Solari/DLSS absorb vs. which stay custom.
- **Full GUI** — `Gui/` ImGui tree (header bar, world/editing/entities/models/debug panels, sky debug). Target only has a one-panel diagnostics HUD; the editor UI is unported.
- **Settings / IO / persistence** — `Settings.cs` + `BuildFlags`, `IO.cs` (input state), `Common/Extensions/File/`, `Common/PathHandler.cs`, `.cvox` world format. App config, input handling, world save/load.
- **Common math/util** — `Common/DataTypes/Point3.cs`, `Common/Cube.cs`, `Common/Helper.cs`, `Common/DynamicStructuredBuffer.cs`. Mostly replaceable by `glam` + Bevy, except `DynamicStructuredBuffer` (a growable GPU buffer abstraction) which needs a `wgpu`/Bevy equivalent.

## Top reuse recommendation

**Bevy Solari is the top reuse target — and it is already wired into the target codebase.** The single most important thing the orchestrator should carry forward is that `bevy-naadf` has *already committed* to using Bevy 0.19's `bevy_solari` (realtime raytraced GI + reference pathtracer) and DLSS Ray Reconstruction as the rendering/denoising substrate (`src/main.rs:20,48-60`, `src/camera.rs:67-78`). NAADF's largest and most fragile subsystem to port would otherwise be its hand-written HLSL ray/GI/resampling/TAA/denoise pipeline (`Content/shaders/render/**`) — and Solari + DLSS-RR already provide functionally overlapping capabilities. The port should therefore be scoped as: **build the voxel + AADF data structures and feed them into Solari** (as `RaytracingMesh3d` BLAS content and/or custom acceleration), rather than reimplementing NAADF's renderer. The surrounding scaffold (`main.rs` app wiring, `camera.rs` Solari/DLSS camera setup + toggle, `hud.rs` diagnostics overlay, the whole Cargo/`.cargo`/toolchain build config) is all directly reusable as-is. The procedural `scene.rs` is the only file that is throwaway placeholder. Everything under "subsystems still entirely unported" — above all the AADF multi-layered cell data structure — is honest greenfield with no existing coverage in the target.
