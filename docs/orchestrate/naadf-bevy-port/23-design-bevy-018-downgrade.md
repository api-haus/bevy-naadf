## §4. API-delta audit — every 0.19 → 0.18 reversal

Walked the full `crates/bevy_naadf/src/` tree breadth-first; the deltas are
listed in **descending load-bearing-ness order**.

### §4.1 BIG ONE: Render-graph nodes are not `Core3d`-schedule systems in 0.18

**The 0.19 model** (current code): render-graph nodes are *systems* in the
`Core3d` schedule, ordered with `.in_set(Core3dSystems::PostProcess)
.before(tonemapping)`. The render context is a `RenderContext` `SystemParam`
plus `ViewQuery<...>` `SystemParam`. Example:
```rust
pub fn naadf_first_hit_node(
    mut render_context: RenderContext,
    pipeline_cache: Res<PipelineCache>,
    pipelines: Res<NaadfPipelines>,
    world_gpu: Option<Res<WorldGpu>>,
    frame_gpu: Option<Res<FrameGpu>>,
) { … }
```

**The 0.18 model** (target): render-graph nodes are `Node`-trait
implementations on a unit struct, registered with
`render_app.add_render_graph_node::<MyNode>(Core3d, MyNodeLabel)` and ordered
with `render_app.add_render_graph_edges(Core3d, (LabelA, LabelB, ...))`.
`Core3d` in 0.18 is a `RenderSubGraph` from `bevy::core_pipeline::core_3d::graph::Core3d`,
**not** a `ScheduleLabel`.

**Replacement pattern** — `Node` (not `ViewNode`, since most nodes are
view-agnostic):
```rust
#[derive(Debug, Hash, PartialEq, Eq, Clone, RenderLabel)]
pub struct NaadfFirstHitLabel;

#[derive(Default)]
pub struct NaadfFirstHitNode;

impl Node for NaadfFirstHitNode {
    fn run<'w>(
        &self,
        _graph: &mut RenderGraphContext,
        render_context: &mut RenderContext<'w>,
        world: &'w World,
    ) -> Result<(), NodeRunError> {
        let pipeline_cache = world.resource::<PipelineCache>();
        let pipelines = world.resource::<NaadfPipelines>();
        let Some(world_gpu) = world.get_resource::<WorldGpu>() else { return Ok(()); };
        let Some(frame_gpu) = world.get_resource::<FrameGpu>() else { return Ok(()); };
        let Some(pipeline) = pipeline_cache.get_compute_pipeline(pipelines.first_hit_pipeline) else { return Ok(()); };

        let workgroups = frame_gpu.pixel_count.div_ceil(FIRST_HIT_WORKGROUP_SIZE).max(1);

        let diagnostics = render_context.diagnostic_recorder();
        let diagnostics = diagnostics.as_deref();
        let encoder = render_context.command_encoder();
        let time_span = diagnostics.time_span(encoder, FIRST_HIT_SPAN);
        {
            let mut pass = encoder.begin_compute_pass(&ComputePassDescriptor {
                label: Some("naadf_first_hit_pass"),
                timestamp_writes: None,
            });
            pass.set_pipeline(pipeline);
            pass.set_bind_group(0, &world_gpu.bind_group, &[]);
            pass.set_bind_group(1, &frame_gpu.bind_group, &[]);
            pass.set_bind_group(2, &frame_gpu.first_hit_atmosphere_bind_group, &[]);
            pass.dispatch_workgroups(workgroups, 1, 1);
        }
        time_span.end(render_context.command_encoder());
        Ok(())
    }
}
```

### §4.2 `GpuResourceAppExt` / `init_gpu_resource` is 0.19-only

**0.19 form** (in `render/mod.rs:124`, `render/construction/mod.rs:2015`):
```rust
render_app.init_gpu_resource::<NaadfPipelines>();
```

**0.18 form** — `init_gpu_resource` does not exist. The 0.18 trait
`SubApp::init_resource::<R>()` works for `FromWorld` resources **but only if
the `RenderDevice` resource already exists** (because `init_resource` calls
`FromWorld` immediately, not in `RenderStartup`).

Per `bevy_render-0.18.1/src/lib.rs:511`, `RenderStartup` is run inside the
render-recovery closure (= after the `RenderDevice` is created). For
`FromWorld` resources that need the `RenderDevice`, the canonical 0.18 idiom
is to `add_systems(RenderStartup, init_resource_fn)` with a helper that runs
once.

Simpler equivalent: define a tiny local replacement at the call site:
```rust
fn init_gpu_resource<R: Resource + FromWorld>(world: &mut World) {
    let res = R::from_world(world);
    world.insert_resource(res);
}
// then:
render_app.add_systems(RenderStartup, init_gpu_resource::<NaadfPipelines>);
```

This works identically — see `bevy_render-0.18.1/src/globals.rs:21`
(`init_resource::<GlobalsBuffer>()` is registered the same way for resources
that don't need `RenderDevice`; for those that *do*, the `add_systems(RenderStartup, ...)`
form is correct).

**Affected sites:** 2 (the two `init_gpu_resource::<X>()` calls). Plus the import
removal `use bevy::render::{… GpuResourceAppExt, …}` in both files.

### §4.3 `Core3dSystems` is 0.19-only

**0.19 form** — `.in_set(Core3dSystems::PostProcess).before(tonemapping)` is
the schedule-system idiom.

**0.18 form** — no `Core3dSystems` enum at all. Render-graph order is
encoded in `add_render_graph_edges`, not in `SystemSet`s. The
`.in_set(Core3dSystems::PostProcess)` constraint is *implicit* in 0.18: render-graph
nodes run in the `Render` schedule's `Render` set (not `PostProcess`), but the
*ordering* relative to `Node3d::Tonemapping` is enforced by the
`add_render_graph_edges` chain ending in `Node3d::Tonemapping`.

So the constraint "before tonemapping" survives as an edge to `Node3d::Tonemapping`
in `add_render_graph_edges`.

## §5. wgpu / storage-texture viability — the GO/NO-GO check

**GO.**

Reasoning:
- Bevy 0.18.1 transitively pins `wgpu = 27.0.1`. Bevy 0.19-rc.1 pins `wgpu = 29.0.3`.
- The Phase-C construction binds `chunks` as `texture_storage_3d<rg32uint, read_write>`.
- The wgpu `Rg32Uint` `guaranteed_format_features` row is identical between
  wgpu 27 (`wgpu-types-27.0.1/src/lib.rs:3182`) and wgpu 29
  (`wgpu-types-29.0.3/src/texture/format.rs:982`):
  ```text
  Rg32Uint =>  (s_ro_wo, all_flags),
  ```
  i.e. **`STORAGE_READ_ONLY | STORAGE_WRITE_ONLY` is in the *guaranteed* feature
  set; `STORAGE_READ_WRITE` is NOT** — both versions require the adapter to
  enable `Features::TEXTURE_ADAPTER_SPECIFIC_FORMAT_FEATURES` (or equivalent)
  for `read_write` to be valid. Since this already works at 0.19, the adapter
  must already expose it; the same will hold at 0.18 because the wgpu source
  is identical for this format row.
- naga storage-access validation does not gate on the wgpu version — naga 27
  (Bevy 0.18) and naga 29 (Bevy 0.19) both accept `StorageAccess::LOAD | STORE`
  for `Rg32Uint` per `naga-27.0.3/src/valid/interface.rs:128`.

**Conclusion:** the §5 risk does NOT block the downgrade. The Phase-C
producer chain should continue to work bit-identically against wgpu 27.

The only residual risk in this layer is a *wgpu-internal* change between 27
and 29 that we haven't audited (descriptor field names, etc.). Those will be
caught by `cargo build`.

## §8. Self-review notes — what changed the plan

### §8.1 The mechanical scope is **much larger than the brief assumed**

The brief lists 11 "known Bevy 0.18 → 0.19 deltas to reverse." The audit
adds:

- **Item 1 (#[require(...)]):** not relevant — `#[require(...)]` was
  introduced *between* 0.18 and 0.19 BUT in `bevy_render-0.18.1/src/view/mod.rs`
  and `bevy_camera-0.18.1/src/components.rs`, `#[require(...)]` is **already
  present in 0.18.1**. So this delta is **not** something to reverse — the
  port's `Hdr`-as-required-component pattern works identically.
- **Item 2 (render graph API):** confirmed — the §4.1 finding above. This is
  by far the largest mechanical lift.
- **Item 3 (World::resource_scope, run_system):** audit found no live use of
  these signatures in the port. Skipped.
- **Item 4 (Component derive):** no live use of `#[component(storage = …)]`
  or `#[component(on_add = …)]` in the port. Skipped.
- **Item 5 (SystemParam trait):** no hand-written `SystemParam` impls.
  Skipped.
- **Item 6 (Schedule / Schedules):** no direct `Schedules` API use.
  Standard `add_systems` works identically.
- **Item 7 (Bevy 0.19-rc-only features):** the `init_gpu_resource` /
  `GpuResourceAppExt` is the main one (§4.2 above). No other 0.19-only
  features found.
- **Item 8 (`bevy_dlss` plumbing):** feature names are identical (verified
  in `bevy_internal-0.18.1/Cargo.toml`).
- **Item 9 (Texture/storage-texture binding APIs):** §5 above — `Rg32Uint`
  storage-texture binding is wgpu-equivalent between 27 and 29.
- **Item 10 (RenderApp::main_mut etc.):** no live use. Skipped.
- **Item 11 (compute pipeline creation):** standard `ComputePipelineDescriptor`
  works identically in both versions.

**Net:** §4.1 (render-graph-node rewrite — 18 nodes) is ~90% of the work.
Everything else is small mechanical adjustments.

## Independent review

Adversarial self-review of the §1–§10 design above.

### Bug 1 — the renderer's view-aware nodes need `ViewNode`, not `Node`

The 0.19 schedule-system render-graph-node form is *view-aware automatically*
because the `ViewQuery` system-param plumbs the active view. The 0.18
`Node`-trait form is **view-agnostic by default** — `Node::run` only has access
to the `World`, not a specific view entity. For view-specific nodes (which
need `ExtractedView`, `ViewTarget`, etc.), the 0.18 idiom is `ViewNode` +
`ViewNodeRunner::<MyViewNode>` registered via `add_render_graph_node`.

**The port's render-graph nodes that depend on view-specific data:**

- `naadf_final_blit_node` — reads `ExtractedView::target_format` (for the
  per-format blit pipeline cache lookup) and `ViewTarget` (for the
  render-pass color attachment).
- The other 17 nodes mostly read non-view resources (`WorldGpu`, `FrameGpu`,
  `TaaGpu`, `NaadfPipelines`, etc.) — these are view-agnostic, so plain
  `Node` works for them.

**Decision:** `naadf_final_blit_node` uses `ViewNode`. The other 17 nodes use
plain `Node`. The §4.1 design above did not split these — fixing now.

**ViewNode example** for the final blit:
```rust
#[derive(Default)]
pub struct NaadfFinalBlitNode;

impl ViewNode for NaadfFinalBlitNode {
    type ViewQuery = (&'static ExtractedView, &'static ViewTarget);

    fn run<'w>(
        &self,
        _graph: &mut RenderGraphContext,
        render_context: &mut RenderContext<'w>,
        (extracted_view, view_target): QueryItem<'w, Self::ViewQuery>,
        world: &'w World,
    ) -> Result<(), NodeRunError> {
        // ... same body, but with view available directly.
    }
}

// register with ViewNodeRunner
render_app.add_render_graph_node::<ViewNodeRunner<NaadfFinalBlitNode>>(
    Core3d, NaadfFinalBlitLabel,
);
```

This is **HIGH RISK** for self-certification — I had it wrong on the first
pass. Carrying forward to the fresh-eyes recommendation list.

releasing the borrow; then `time_span.end(...)` re-borrows. This pattern is
canonical in 0.18. **No change.**

## Decisions & rejected alternatives

1. **Chose:** Pin `bevy = 0.18.1` (latest stable 0.18.x patch).
   **Rejected:** `bevy = 0.18.0` (earlier patch). **Why:** 0.18.1 is the latest
   patch, includes any 0.18-line bug-fixes. **What would flip:** if `bevy_egui`
   pins to `=0.18.0` exactly (not `^0.18`), we'd need 0.18.0. Verified
   `bevy_egui 0.39.1` declares `bevy_app = 0.18.0` which is `^0.18` —
   compatible with 0.18.1.

2. **Chose:** Replace `init_gpu_resource::<X>()` with
   `add_systems(RenderStartup, init_gpu_resource_fn::<X>)`.
   **Rejected:** Replace with `init_resource::<X>()` directly.
   **Why:** `init_resource` calls `FromWorld` immediately, but
   `NaadfPipelines::from_world` and `ConstructionPipelines::from_world` both
   need the `RenderDevice` resource, which only exists after
   `RenderPlugin`'s render-device-acquire step. Calling
   `init_resource` in `Plugin::build` would panic. The `RenderStartup`-system
   form is the canonical 0.18 idiom for this. **What would flip:** nothing
   — this is mechanical.

3. **Chose:** `naadf_final_blit_node` as a `ViewNode`; the other 17 as plain
   `Node`s. **Rejected:** all 18 as `Node`. **Why:** `naadf_final_blit_node`
   reads `ExtractedView::target_format` and uses `ViewTarget` for the
   render-pass color attachment — these are per-view data the `ViewNode`
   pattern delivers natively. The other 17 read non-view resources. **What
   would flip:** if there's a node I missed that needs `ExtractedView`, it
   would need `ViewNode` too. Audit candidate: search every node body for
   `ExtractedView` / `ViewTarget` access.

4. **Chose:** Drop the `.in_set(Core3dSystems::PostProcess)` constraint
   silently (no 0.18 equivalent). **Rejected:** Try to mimic it. **Why:** in
   0.18, render-graph nodes run inside the `Render` schedule, and the
   `PostProcess` notion is structurally absent. The functional equivalent —
   "after the main pass, before tonemapping" — is encoded in the
   `add_render_graph_edges` chain ending at `Node3d::Tonemapping`. Per the
   alignment-gap doc, the port is a compute-only pipeline that does its own
   raytracing and doesn't depend on Bevy's main-3D-pass output, so the
   "after main pass" implicit constraint is moot — only the
   "before tonemapping" matters, and that's covered by the edge chain.

5. **Chose:** `Hdr` from `bevy::render::view::Hdr`. **Rejected:**
   `bevy::camera::Hdr`. **Why:** 0.18 has `Hdr` in `bevy_render::view`, not
   `bevy_camera`. The marker semantics are identical.

6. **Chose:** Strip the `bevy::core_pipeline::tonemapping::tonemapping`
   import + the `.before(tonemapping)` use. **Rejected:** Keep them.
   **Why:** the `tonemapping` *function* doesn't exist in 0.18; only the
   render-graph *node* `Node3d::Tonemapping` does. The "before
   tonemapping" constraint moves into the `add_render_graph_edges` chain.

7. **Chose:** Refit `TextFont { font_size: 14.0, ..default() }` (raw `f32`).
   **Rejected:** Keep `FontSize::Px(14.0)`. **Why:** `FontSize` enum doesn't
   exist in 0.18; the field is plain `f32`.

8. **Chose:** Implement the impl in one cycle (Cargo → mechanical → big
   rewrite → iterative cargo build → tests → e2e). **Rejected:** Break into
   sub-phases. **Why:** consolidated-mode dispatch — no checkpoint pauses.
