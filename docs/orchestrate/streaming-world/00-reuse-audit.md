## Gaps — what does not exist yet

- **No residency manager.** Zero code anywhere in `crates/` for "currently-resident chunks", "active world region", "chunk indirection table", "LRU eviction", "VRAM budget tracking", "ring buffer of chunk slots", or "stream in/out". Grep for `stream | residen | sliding | lru | evict | active.region` returned only TAA / pipeline-stream / sample-stream hits — none about world residency. Implication: the residency layer is greenfield.
- **No procedural noise feeder wired into the renderer.** `voxel_noise` is a sibling crate with **no `[dependencies]` entry in `crates/bevy_naadf/Cargo.toml`**, no `use voxel_noise::…` anywhere in `crates/bevy_naadf/src/`. Implication: wiring is one Cargo edit + a new module; nothing is fighting for that seam.
- **No noise→ModelData adapter.** The W5 `ModelData` resource takes a *pre-encoded* three-layer model. There is no helper that samples a `voxel_noise::NoiseNode` over a chunk extent and produces the encoded `data_chunk`/`data_block`/`data_voxel` for that chunk. Implication: either (a) write a noise→ModelData helper, or (b) write a noise→DenseVolume→`construct()` helper, or (c) bypass `ModelData` entirely and write directly into the segment voxel buffer via a new compute shader.
- **No per-chunk encoding seam.** Both `aadf::construct(&DenseVolume)` and `vox_import::build_constructed_world_sparse` produce one *whole-world* `ConstructedWorld`. To stream one chunk at a time, the inner per-chunk encoder body (classify → dedup → emit blocks → emit voxels) needs to be lifted into a public function. Implication: localised refactor of `aadf/construct.rs` and/or `voxel/vox_import.rs`.
- **No coordinate-system widening for "large worlds".** Everywhere world-space is addressed it is `UVec3 chunks` (`size_in_chunks: UVec3`) or `IVec3 voxels` (`bounding_box: IAabb3`), with shader-side packing of `(cx:11,cy:10,cz:11)` bits in `world_change.wgsl` and `edit.rs:330-332` — capping chunk coordinates at `2048×1024×2048`. Implication: "infinite worlds" with coordinates past that needs either a chunk-pos repacking shader change OR a per-window-relative coordinate scheme where the GPU only ever sees local-window indices.
- **No wasm web-worker thread pool helper crate.** The user's brief mentions "a helper crate that bridges fastnoise2 on web with a webworker thread pool". The `voxel_noise` JS bridge supports being initialised in a worker context (`voxel_noise_bridge.js:20` detects `WorkerGlobalScope`), but no Rust-side pool/dispatcher crate exists. The `workers/` dir is a single Cloudflare R2 proxy (`workers/r2-proxy/wrangler.toml`), unrelated to compute workers. Implication: if a parallel-web-worker generator is part of the design, that crate is greenfield.
- **No partial-write / write-region API on `WorldGpu`.** `GrowableBuffer::reserve` grows by 2× + full copy; `upload_all` writes from index 0. The only existing "write into existing storage at offset" path is the W2 GPU dispatch (which atomic-cursor-appends, not random-access). Implication: streaming writes go through W2 record buffers OR a new `RenderQueue::write_buffer(buf, byte_offset, &chunk_data)` system.
- **No "ingested chunk source abstraction".** The two existing producers (`construct`, `vox_import`) hardcode their inputs (`DenseVolume`, `dot_vox::DotVoxData`). There is no `trait ChunkSource { fn produce(&self, chunk_pos: IVec3) -> EncodedChunk; }` seam, so swapping in a noise source vs `.vox` source vs Minecraft converter is currently three independent install paths in `voxel/grid.rs::setup_test_grid`. Implication: a uniform "chunk source" trait is needed if all three formats are to share the streaming layer.

## Borderline calls

- **`GrowableBuffer<T>` → reuse vs not-applicable.** Marked **Med confidence** because the brief asks for "sliding-window into fixed VRAM" — growth is exactly the *wrong* primitive for that ("fixed" implies no `reserve()`). What flips this to "extend": if the design carves out the streaming buffer as a *separately allocated, never-grows* `Buffer` with an internal slot table on top, `GrowableBuffer` is reusable for everything *else* (voxel-types palette, residency-metadata) but not for the residency slab itself. If the design opts to size the streaming slab once at startup from a `VRAM_BUDGET_MIB` knob (the natural shape), `GrowableBuffer` is not applicable for the slab.
- **`aadf::construct(&DenseVolume)` per-chunk extraction → extend vs greenfield.** Marked **Med confidence** because the function is currently written as one tight whole-world pass with a single dedup-HashMap shared across all chunks. Per-chunk encoding without that shared HashMap loses block dedup *across* chunks, which materially changes the buffer-size economics on a multi-biome streamed world. What flips this: if the design accepts per-chunk-local dedup only (loss of cross-chunk dedup) the lift is small; if it requires global dedup across an unbounded set of chunks streaming in and out, the existing function is *not* the right starting point — that's a content-addressable storage problem and the `BlockHashingHandler` (`aadf/block_hash.rs`) is closer.
- **`vox_import` per-chunk extraction → extend vs greenfield.** Marked **Med-High** because `ChunkBuckets` already represents the sparse world as one `Option<Bucket>` per chunk, but the dedup HashMap is shared across chunks in `build_constructed_world_sparse`. Same flip condition as above: per-chunk-local dedup is a localised lift, global dedup is a re-architecture.
- **W5 segment loop → extend vs not-applicable.** Marked **High confidence** for the *iteration pattern*, but borderline for *reuse-as-is* because: the loop currently runs **once per app lifetime** (gated by `gpu_producer_has_run`), generates the *entire* `WORLD_SIZE_IN_VOXELS = 4096×512×4096` world at startup, dispatches 512 generator+chunk_calc submits, then never runs again. To repurpose it for sliding-window residency the gate has to invert: per-frame, the system has to dispatch only the segments that newly entered the residency window. That's a significant control-flow rework even though the buffer layout, bind groups, and dispatch shape are reusable verbatim.

## One-line summary

The `voxel_noise` crate, the W5 `ModelData`+`generate_segment_cpu`+`generator_model` segment-iteration machinery, and the W2 delta-upload chain together already cover ~70 % of what a "noise → per-chunk encode → upload to a fixed segment buffer" pipeline needs; the load-bearing greenfield piece is the **residency manager** (which chunks are resident, an `IVec3 chunk-pos → slot-index` indirection table, eviction on window-shift, a VRAM-budgeted slab carved out of `WorldGpu`'s existing fixed buffers).
